# Versioning Strategy

This project follows a simplified Semantic Versioning strategy.

Version format:

> MAJOR.MINOR.PATCH

Example:

> v0.1.0

## Version Meaning

### MAJOR

Represents a significant milestone.

Examples:

- v1.0.0 — Complete portfolio project
- v2.0.0 — Major redesign

### MINOR

Represents the completion of a major learning phase or sprint.

Examples:

| Version | Completed Phase |
|----------|-----------------|
| v0.1.0 | Repository & Project Setup |
| v0.2.0 | Business Analysis |
| v0.3.0 | System Architecture |
| v0.4.0 | API Contracts |
| v0.5.0 | First Frends Process |
| v0.6.0 | Process Development |
| v0.7.0 | Integrations |
| v0.8.0 | Monitoring & Deployment |
| v0.9.0 | Certification Ready |
| v1.0.0 | Final Portfolio Project |

### PATCH

Represents small improvements or corrections that do not introduce a new learning milestone.

Example:

> v0.3.1

# Commit Messages

The project follows Conventional Commits.

Examples:

docs: add project charter

docs: update business requirements

feat: implement receive donation process

refactor: simplify process structure

fix: correct API contract

# Release Strategy

A new Git tag is created only after completing a major project milestone.

Example:

Repository Setup

↓

README

↓

Project Charter

↓

Release
